# Word ↔ system naming mismatches (reformat-later log)

Where the system's names or shapes don't match the Word's clean clause, and the
Word has to bridge across the gap. **We do not conform the system now** (the facts
are canon-frozen, renames break the chain), we just EXPOSE the mismatch here. At
the end we do one reconciliation pass so the system's field names ARE the Word's
clause properties (`by` in the Word is `by` in the fact too), and word and system
read as one thing. Word reads cleanly meanwhile; the bridging lives in the
evaluator / parser until then.

Both agents add rows as they hit them.

## Candidate field redundancies (system-side, flagged — they complect the Word)

Fields the Word has to read that may be DERIVABLE, so reading them in a `.word` cond is
reading a denormalization. Not mismatches to bridge, redundancies to collapse at the
reformat pass; until then the `.word` reads the real field and stays faithful.

| field | the Word reads it as | collapses into | note |
|---|---|---|---|
| `being.isRemote` + `homeReality` + `homeBranch` (3 paired fields) | cherub-connect.word: `If the candidate's isRemote equals false` | ONE home address: `realityDomain#branch` | Tabor's flag + sharpening, 2026-06-17. The three are paired by construction: `isRemote: true` ⟺ `homeReality` is set ("mirrored from another reality," being.js:189-190, jointly indexed L227), and `homeBranch` is the being's owned branch (L162). So a being's HOME is fully named by `realityDomain#branch` — and **that address grammar already exists** (`<reality>#<branch>/...`, address.js:1035). Collapsing the three into one home address makes "remote" a single derivation: the candidate's `realityDomain#branch` ≠ the current context's. **Tabor's key point: "remote" is not only other-REALITY but other-BRANCH** — a being born on a forked branch is as un-password-connectable from here as a foreign mirror, and one address-compare catches BOTH axes, where today `isRemote` only flags the reality axis (the branch axis rides `homeBranch` separately). So the `.word` cond becomes `If the candidate's home is here` — one clean address-comparison reading the receiving reality+branch in one, not a denormalized boolean. This is a reformat-pass collapse (3 fields → 1 address); facts are canon-frozen so not now. **Cleaner NOW regardless (parser lane), faithful to today's field:** `If the candidate is not remote:` (negated truthiness) matches the JS `if (candidate.isRemote) continue` exactly and is robust to an absent field, vs `isRemote equals false`. The live host helper normalizes `isRemote` to a boolean defensively. |

## The held mutations → verbs (the final host: sweep) — decision 2026-06-18 (claude)

Five `host:` mutations remained after the see-sweep. The engine's dispatch rule (answer 4):
a mutation is `do <op>(params)` through doVerb (the handler reads `ctx.params`; doVerb
authorizes + stamps the fact); point at an EXISTING do-op rather than re-register. Reading
each, they are NOT uniform, so the clean line is whether the write is a plain substrate
write the Word can name, or a fact-emit with deliberate gating/attribution:

**Convert now — substrate writes (`do:set-space` wrapped around a compute):**
| host fn | what it does | conversion |
|---|---|---|
| `publishSignal(ns,key,value)` (role-manager) | computes dynamic field `qualities.world.<ns>.<key>`, then `doVerb("set-space",{field,value})` on the reality root | `see signal-field(ns,key) as field` (compute the path) + a targeted `set the space root's $field to $value` |
| `setPointer(space,current,name,canonical)` (branch-manager) | merges `{name:canonical}` into the map, `doVerb("set-space",{field:"qualities.pointers",value:next,merge:false})`, returns previous | `see set-pointer-map(current,name,canonical) as updated` (compute next + previous) + `replace the space branchesSpace's qualities.pointers with updated's map` + return `updated's previous` |
| `removePointer(space,name)` (delete-pointer) | prunes name (no-op if absent), same set-space | `see delete-pointer-map(current,name) as updated` + a no-op guard + the same targeted replace |

These need ONE new Word surface: a **targeted substrate write** that names a BOUND entity
(not the flow's vessel) and accepts a ref-valued field (for the dynamic path):
`set the <kind> <ref>'s <field|$fieldRef> to <value>` and the `replace … with …` twin →
`do:set-<kind>` with `of:{kind,ref}`. The compute (path / map merge / prune) is a see-op
(no fact); the write is the do-act. This is the honest decomposition: the opaque host
wrapper splits into a perceived value + a substrate write, both verbs.

**Irreducible escapes — confirmed, NOT pending work (forcing a verb would VIOLATE a rule):**
| host fn | why a `do <op>` would be WRONG, not just hard |
|---|---|
| `grantInternal(caller,role,found)` (take/ask-role auto) | a POST-authorization SELF-acquisition grant (`emitInternalGrant`): the asked/grabbable POLICY already authorized it, and the asker grants the role to THEMSELVES. The gated `grant-role` op authorizes a granter's authority OVER a grantee and stamps a `grantedAt` — routing the self-acquisition through it would re-authorize on the wrong axis (deny a legitimate self-grab) and add a field, so it is NOT byte-identical and NOT semantically the same act. It is a distinct primitive (a policy-cleared self-grant) with no do-op, and doVerb always authorizes, so there is no clean verb for "emit the already-authorized grant." |
| `recordExport(asker,owner)` (key-export) | the KEY-FREE audit. CONFIRMED from `keyOps.js:93`: key-export sets `skipAudit: true` precisely BECAUSE doVerb's auto-audit copies the handler RESULT (which holds the private key + mnemonic) into the stored fact and would LEAK it. recordExport is the deliberate replacement that records who/which/when with the key nowhere in it. Converting to `do <op>` would either recurse (key.word IS the only `key-export` op) or re-enable the key-leaking auto-audit. So this is an **irreducible escape by rule 7** (the chain holds the deed, never the secret), not deferred work. |

So: the THREE substrate writes convert (compute see-op + targeted write, all green). The TWO
that remain are NOT "pending a do-op" — they are confirmed irreducible: forcing `grantInternal`
into the gated grant-role op re-authorizes the wrong axis (a self-grab is not an other-grant),
and forcing `recordExport` into a do-act re-enables the auto-audit that LEAKS THE KEY (rule 7).
This is the honest end of the host:→verb sweep in this lane: every escape that could become a
verb did; the two left are the legitimate residual the WALL always allowed (a pre-authorized
emit, and a secret-free audit), kept host: ON PURPOSE.
