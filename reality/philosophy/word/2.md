# The Word: build plan (now to a frozen, host-agnostic world)

This plan takes the in-world layer of TreeOS out of JavaScript and into `.word`, builds and shapes the language until it is acceptable, converts the world to it slice by slice, and freezes the result as a specification. After this plan, the world is a **shared, host-agnostic language**: the engine that runs it stops mattering.

It is written so a builder with no prior context can pick it up and continue. Read [1.md](./1.md) for what The Word is and [0.md](./0.md) for the long derivation.

## Context for a new builder

**TreeOS in one paragraph.** TreeOS is a substrate where a "reality" is made entirely of facts. The nouns are space (place), matter (thing), and being (presence). A **Name** is the cryptographic identity that signs; a being is the body a Name acts through; a soul is the cognition that decides. Five verbs reach the substrate: SEE, DO, SUMMON, BE act in the world, and NAME mints identity outside it. Every act runs through the **stamper**, a five-beat moment cycle (intake, assign, fold, momentum, stamped). Facts are content-addressed: a fact's id is the hash of its content, so the chain is replayable and every host must produce identical hashes from identical inputs. State is never stored as a final value; it is **folded** from the facts behind it.

**What The Word is.** The role, act, flow, type, and seed layer (what a reality's behavior IS) is currently imperative JavaScript: `can-see` / `can-do` / `can-summon` functions, operation handlers, roleflows, subscriptions, matter-type registries, seeds. The Word replaces all of that with word-shaped declarations, literally constrained English sentences ("A dj can queue a song to the playlist.") that are also data the substrate evaluates. Code becomes substrate data: queryable, composable, federable, self-describing. See 1.md.

**Where the code lives (current tree).**
- `reality/seed/` is the kernel.
- In-world today (the migration targets), everything an agent does: `seed/present/roles/*` (per-role can-see/can-do/can-summon and roleFlow: cherub, harmony, federation-manager, birther, auth, and the rest), the DO operation registry under `seed/ibp/` (`registerOperation` and handlers), the operation rules of the other verbs (BE's birth/connect/release/switch/death; NAME's declare/banish in `seed/ibp/nameOps.js` and `seed/ibp/verbs/name.js`), I_AM's own root acts (genesis, declaring names, signing roots), see-op shaping in `seed/ibp/descriptor.js`, permission logic in `seed/ibp/roleAuth.js` and `authorize.js`, flows in `seed/present/wakes/` (`subscriptions.js`, `wakeSchedule.js`) and the `core.declare.*` surface (subscribe, schedule, aggregate), matter/space/being type declarations under `seed/materials/`, and seeds under `seed/materials/publish/`. The verb dispatch, transports, and session channels that carry these acts stay host.
- Host (stays in the host language): `seed/present/stamper/*` (1-assign, 2-fold, 3-momentum, 4-stamped), `seed/past/fact/*` and `seed/past/act/*` (facts, hash, chainRoots, actChain, actSig), `seed/materials/projections.js` (the fold), storage (Mongo today), `reality/transports/*`, crypto and Peering, and the Word parser and evaluator themselves.

**Current state (read before starting).** The substrate is mid a Name / Being / Soul refactor (identity split into Name + Being + Soul; see `philosophy/names/names.md` and the factory `name` and `identity` pages). The wire and stamper recently renamed the operation selector to `act` end to end (an act in flight; only the sealed fact records it as `action`). The Word build should be designed against the substrate AFTER these primitives stabilize, not against a moving target. Canon vectors (`reality/.test/scripts/verify-canon-vectors.js`) freeze fact hashes; keep them green.

## What this plan delivers, and why it must be a shared language

**Delivers:** the in-world layer extracted into `.word`, an interpreter for it in the current host, a surface grammar you have signed off on, the full world converted with the system live throughout, and a frozen conformance suite that defines the world's behavior in `.word` rather than in JavaScript.

**Why a shared language (the downstream reason, not part of this plan).** Once the world is `.word` and its behavior is pinned by a frozen suite, the host that runs it becomes replaceable. Down the line the full OS will run a different host (Rust) on the same, unchanged `.word`. That port is a separate, later effort. It is the reason this plan exists, the reason the world must be a shared language now, but it is out of scope here. The same `.word` has to mean the same thing to:

- **any host** (the JS host today; whatever runs the OS later),
- **every reality** (federation: peers install and run each other's `.word` and must agree),
- **any cognition** (a human and an LLM write the same declarations).

That is what "shared language" means, and it is why the discipline below (a host-agnostic IR, determinism, a conformance suite) is worth paying for now even though only one host exists today.

## The invariant everything serves

> The `.word` corpus, its IR, and a conformance suite are the contract. Any host that parses the IR and passes the suite runs the same world. The same `.word` yields the same chain.

**Determinism is non-negotiable, and it is already required.** Because facts are content-addressed and the chain is replayable, the JS host itself must reproduce identical hashes on replay. The Word evaluator must therefore be deterministic: fixed ordering, fixed number handling, no wall-clock or randomness inside evaluation (those enter as signed inputs, never as evaluator behavior). This is not a Rust concern; it is a chain concern that the evaluator inherits today. It is also exactly what makes a future host trivially correct.

## The cut: world versus host

The cleanest statement of the boundary: **the world is everything an agent does.** I_AM, every Name, and every being act; what they perceive (SEE), do (DO), reach (SUMMON), become (BE), and declare as identity (NAME) is `.word`, down to I_AM's own genesis and signing acts and the NAME-layer ops (declare, banish). The substrate's root behavior is word too; this is the meta-circular property. The **host** is the engine that runs those acts and keeps the chain. It holds no opinion about any particular reality; it just executes what agents declare.

A second test, equivalent, for any current `.js`: **would two different realities differ here?**

- Reality-specific behavior an agent or community could write or federate (a role's can-see, an op, a roleflow, a flow rule, a matter type, a seed, a Name's identity ops) is **world**; it becomes `.word`.
- The same engine for every reality (intake, assign, fold, momentum, stamped, the fact chain, hashing, signing, projection, storage, transports, the rate limiters and session channels, the Word parser and evaluator) is **host**; it stays in the host language.

**The edges decompose cleanly.** Even an agent act that rides host plumbing splits in two. A NAME declare is a Name's act, so its meaning and rules are `.word`; the socket channel, rate limiter, session binding, and key crypto that carry it (`protocols/ibp/nameSession.js`) are host. I_AM's genesis is an act (word); the bootstrap mechanism that opens it (`withIAmAct`, the stamper) is host. The WHAT is always word; the HOW is always host.

The **escape hatch** is the seam where a word act reaches into host computation: a declaration names a host builtin. The builtin is host code; the declaration is world. Keep the deterministic builtins small and specified by vectors; keep effectful compute (network, ML, external IO) clearly outside the replayable core, where the foreign-by-construction doctrine already governs it.

## Phase 0: boundary and inventory (the cut list)

Before code, produce the backlog: walk the in-world directories above and tag every site "becomes `.word`" or "stays host," tracked to zero. Output a single checklist file and an explicit written statement of what the host is. This is the spec for Phase 5 and the definition of the host boundary.

## Phase 1: the IR (the host-agnostic contract)

Define the AST / intermediate representation before the surface grammar. The IR is the data a `.word` program parses into and that the evaluator runs. The surface can evolve; the IR is the stable contract.

- Plain, JSON-serializable nodes with explicit type fields (role-declaration, act, flow, can-see expression, condition, effect, fact, compute-call).
- Content-addressable, so a compiled program has a stable id and federates like any resource.
- Versioned, so grammar changes are explicit and the conformance suite can detect drift.

Gate: the IR can represent the hardest existing slice (chosen next) with no opaque host escape beyond named compute calls.

## Phase 2: evaluator first (semantics before syntax)

Pick one real, hard slice: the cherub register flow, or the harmony drummer plus dancers (these exercise roles, acts, a roleflow, and a flow at once). Hand-build its IR and write the evaluator that runs it inside the existing stamper. No parser yet.

Gate: the evaluator reproduces the current JS handlers' behavior on that slice, proven by diffing facts and folds against a pre-change run. Get the semantics right before committing to syntax.

## Phase 3: find the shape (the acceptance gate)

This is the checkpoint that matters most. Lock a surface grammar that is acceptable before any sweep.

- Build the word-shaped surface and a hand-written recursive-descent parser that emits the Phase 1 IR.
- Re-express several real TreeOS pieces in `.word`: a role with real can-see/can-do, a real roleFlow, the cherub register flow, a harmony seed.
- Iterate against what is awkward or missing (pronouns and variable binding, quantifiers, articles, adverb-style act modifiers were already flagged in 0.md and 1.md).

Gate (human, Tabor): read the `.word` versions of real logic and accept the shape. Lock a v0 grammar spec and v0 IR. Phase 5 does not start until the shape is signed off; converting the whole world to a shape you do not like is the expensive mistake.

## Phase 4: the bridge (coexistence)

Make a role, op, or flow be either a JS handler or a `.word` declaration, with the stamper resolving both, so the world can migrate slice by slice while live.

- The registries become dual: handler-or-declaration, preferring the `.word` form when present.
- Specify the escape-hatch contract concretely: how a `.word` act names a host builtin, how arguments and results cross the boundary, and the sandbox. Split builtins into the deterministic standard library (vector-specified) and effectful compute. Start the standard-library vectors here.

## Phase 5: conversion sweep (in-world JS to .word)

Work the Phase 0 backlog, ordered by leverage, each slice verified against its pre-conversion facts so the e2e suite stays green:

1. **See-op dissolution plus roles** first: roles declare their own reads, the see-registry disappears, depSet (which reels were read) becomes automatic from the patterns. Biggest simplification.
2. **DO operations** to act declarations, with compute calls where real computation remains.
3. **Roleflows and flows** (subscriptions, schedules, aggregates) to declarative form.
4. **Type declarations** (matter types, space templates, being shapes).
5. **Seeds** to `.word` seed programs.

Every converted slice deletes its JS handler; the host shrinks as the world grows. Track to "zero in-world JS remaining." Do not begin the sweep until the current refactor has landed.

## Phase 6: the conformance suite (the frozen spec) — end of this plan

Assemble a corpus of `.word` programs with expected outcomes (facts, folds, descriptors), frozen the way the canon vectors are frozen. The host must pass it. From here on, the world's behavior is defined by `.word` plus this suite, not by JavaScript.

At this point the deliverable is complete: the world is a shared, host-agnostic language with a specification. The engine underneath is now replaceable, which is the whole point.

## Out of scope (downstream, enabled by this plan)

- **The host port to the OS runtime (Rust).** A later, separate effort: reimplement the engine against the unchanged `.word` corpus and prove parity by passing the canon vectors and the Phase 6 conformance suite, then run both hosts on the same `.word` and confirm byte-identical facts. This plan exists so that port is "rewrite the engine," never "rewrite the world." It is the reason for the shared language, not a step in building it.

## Sequencing against current work

- **Now, alongside finishing the refactor:** Phase 0 (cut list) and Phase 1 (IR). These clarify the substrate's own boundary and do not depend on the surface grammar.
- **As the primitives settle:** Phase 2 (evaluator on one slice) and Phase 3 (the shape, the acceptance gate).
- **After the refactor lands and the shape is locked:** Phases 4 to 6 (bridge, sweep, conformance), the system live throughout.

## Risks held honestly

- **Determinism** is the deepest requirement, and it bites the JS host today via replay and CAS, not only a future host. Pin number handling, iteration order, and evaluation order in the v0 IR spec, and test with replay vectors early.
- **The escape hatch** is where host-agnosticism is hardest. Keep the deterministic standard library small and vector-specified; keep effectful compute outside the replayable core.
- **Grammar churn** after Phase 3 is the expensive failure. The acceptance gate plus a versioned IR contain it: surface changes recompile to the same IR, and the conformance suite catches drift.
- **Performance:** the evaluator runs on every moment for perception and authorization. Design caching, incremental evaluation, and indexing into the evaluator from Phase 2.

The shape of the whole plan: extract the world into `.word`, prove it with a frozen suite, and the host language stops mattering. Building that shared language is this plan; swapping the engine underneath it is the future it unlocks.
