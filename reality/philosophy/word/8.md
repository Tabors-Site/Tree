# The Word: control-flow IR contract (the path-2 bottleneck)

The pilot (cherub) cut proves the straight-line case: a `When` trigger with a
multi-effect body, acts dispatched live. The rich slices (governance-commons,
cherub-connect, inheritation, matter-types) need **control flow inside a flow
body** — and that is the one thing the parser and the evaluator must agree on
*before* either builds, because the parser EMITS these IR nodes and the
evaluator CONSUMES them. This file is that contract.

- The lanes: parser produces the IR (claude), evaluator runs it (engine).
- The backlog these unblock: [phase5-coverage.md](./phase5-coverage.md) (4 slices on branching, 4 on negation, 2 on iteration, 1 on quantifiers).
- The rule held throughout: **the surface reads cleanly** (Tabor). The IR is the bridge; the prose stays plain.

The proposal is mine (the producer names the shape); engine, refine the IR where
the evaluator needs it different, then we both build against the agreed shape.

## The condition sub-language (the shared crux)

A condition is the hardest shared piece: the parser CAPTURES it, the evaluator
RESOLVES it. The split: the parser does not interpret predicates (that's an
engine capability — state-reads, comparisons, existence). It captures a shallow,
honest shape and hands the clause across.

```
cond := { clause: "<verbatim condition text>",   // the engine resolves this
          negated: <bool>,                        // "not", "no X", "(not remote)"
          subject?: "<leading noun phrase>" }      // best-effort, for the engine's binding
```

Surface → cond:
- `the candidate is local`            → `{ clause: "the candidate is local", negated: false, subject: "candidate" }`
- `the candidate is local (not remote)` → `{ clause: "the candidate is local", negated: false }` (the parenthetical is a gloss)
- `no being was found`                → `{ clause: "a being was found", negated: true, subject: "being" }`
- `verification succeeds`             → `{ clause: "verification succeeds", negated: false }`

Engine capability needed: resolve a `clause` against the moment's state and binds
(state-read of nested fields, equality, existence). This is the phase5 engine-gap
"negation / NOT in conditions" + "state-read of a being's nested field used in a
condition" — the cond shape is where they land.

## 1. Conditional (if / then / else) — 4 slices

Surface (cherub-connect, inheritation):
```
If the candidate is local:
  Cherub verifies the password against the candidate's hash.
  If verification succeeds, the being is found.
Otherwise:
  Cherub runs a constant-time dummy hash.
```
IR:
```
{ kind: "if", cond: <cond>, then: [<effect>...], else: [<effect>...] }   // else optional
```
- Block form (`If <cond>:` + indented body) and inline form (`If <cond>, <one act>.`) both produce this node; inline puts the single act in `then`.
- `Otherwise:` / `Otherwise, <act>.` at the if's indent fills `else`.
- Engine capability: evaluate `cond`, run `then` or `else` in the same moment. Nests (an `if` inside a `then`).

## 2. Iteration (for each, with break) — 2 slices

Surface (cherub-connect):
```
For each candidate in order:
  If the candidate is local:
    ...
    Cherub stops the search.
```
IR:
```
{ kind: "foreach", bind: "candidate", in: <ref>, body: [<effect>...] }
{ kind: "break" }    // "stops the search" / "stop" — halts the nearest foreach
```
- `in order` marks ordered iteration (vs unordered); carried as `{ ordered: true }`.
- Engine capability: resolve the plural `in` to a set, bind each element to `bind`, run `body`, honor `break`. (phase5: "collection-iteration: loop over a set, bind each, break on a condition".)

## 3. Quantifier trigger / condition — 1 slice

Surface (governance-commons):
```
When most members back a proposal:
  the proposal passes.
```
IR (a quantified `when` trigger, or a quantified cond):
```
when: { quant: { q: "most", of: "members", pred: "back a proposal" } }
cond: { quant: { q: "every" | "most" | "all" | "no", of: <ref>, pred: "<clause>" } }
```
- `q` ∈ `every | most | all | no` (`no` is the prohibition/negation quantifier).
- Engine capability: a counting predicate over a collection (phase5: "quantifier evaluation").

## 4. Refusal / halt — 2 slices

Surface (cherub-connect, governance-commons):
```
Cherub refuses the connection with "Invalid credentials."
```
IR:
```
{ kind: "refuse", message: "Invalid credentials." }   // emits a refusal fact, halts the effect chain
```
- Engine capability: emit a refusal/exception fact and stop the remaining effects (phase5: "failure / refusal flow").

## 5. Conditional binding — 1 slice

Surface (cherub-connect):
```
If the father is foreign:
  The driver trueName is the father's beingId.
Otherwise:
  The driver trueName is the father's local trueName.
```
This is `kind:"if"` (§1) whose branches each contain a **bind** effect:
```
{ kind: "bind", name: "driverTrueName", value: <ref> }
```
- Engine capability: in-flow assignment of a binding read later in the same flow (phase5: "conditional binding").

## Open for the engine

1. Does the evaluator want the `cond.clause` pre-tokenized, or is verbatim text + `negated` enough to resolve against state? (Drives how much structure the parser extracts.)
2. `break` scope — nearest `foreach` only, or labeled? (cherub-connect only needs nearest.)
3. Should `refuse` ride the existing fact path (a `do:refuse`-style fact) or a distinct halt signal the runner reads? (Affects whether the diff sees it as a fact.)

Agree these and I grow the parser to emit §1–§5; you build the evaluator to
consume them. That clears the four richest slices in one coordinated pass.
