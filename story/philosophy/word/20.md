# 20 — Chain-native programming (remove time from the logic too)

_Tabor's before-bed idea, + an expansion. DIRECTION / doctrine, not a task._

## The idea (Tabor)

If we build a full OS and **remove time**, how does the _coding logic that backs the Word_ hold
up — all its loops and control flow? That logic is classic time-based programming (for-loops,
sequential steps, mutable state evolving over a clock). I want to make **all of that the Word**,
and have the logic depend on the **chain** instead of the weird time-logic stuff. The effect would
be like **rebuilding programming from the ground up.**

## The mismatch it names

The _surface_ (the Word) is already timeless and chain-native — facts chain by `on` (the parent's
address), not a clock; "the chain is the clock" (5d.md); a being is a moment (presentism); state is
the **fold** of facts, not a variable mutating over steps. But the _substrate_ under it — the host
JS that implements the Word's machinery — is pure von Neumann time: a program counter, a loop that
steps, memory that mutates. So the timeless thing rides on a time-based engine. Remove time at the
OS level and that engine becomes the odd organ — the "weird time logic" Tabor means.

## Chain-native programming, concretely (the expansion)

Replace **time** (program counter + mutable state over steps) with **causal order** (the chain +
the fold). Each classic construct has a chain-native form that already exists in TreeOS pieces:

- **A loop is a fold that grows the chain.** Each iteration is a _fact_ chaining off the last; the
  "next" is laying the next fact; the program counter is the **chain head**. The loop ends when a
  condition read from the fold holds — and that condition is a **noun-read** (`see` = inert, a live
  sensor on the fold, 17.md), while the body is the **act** (lay the next fact). Generation _is_
  iteration (llm.md, one-fact-at-a-time).
- **State is the fold, never a variable.** Every "assignment" is a new fact; the current value is
  the fold (place-is-folded-from-facts). No mutation, so no aliasing, no stale reads — the value at
  a moment is exactly what the chain folds to there.
- **Control flow is the conditional Word.** `If cond, then …` already is one; branching is a
  **history fork** (a branch in the chain); recursion/iteration is the chain _growing_ — the call
  stack IS the chain.
- **A program is a Book.** Words + the chain of its run = a reel; content-addressed, forkable
  (branch a history = fork a computation), verifiable (`verifyReel`), shareable (the Library). A
  paused program is a chain head; resume = lay the next fact. Persistence is free (the store is CAS).

## Why it's "rebuilding programming from the ground up"

The model stops being _time-native_ (clock, counter, mutable memory) and becomes
**causality-native** (chain, fold, content-addressed facts). What that dissolves:

- **Replay + audit for free.** Every run is a fact-chain → deterministically replayable and
  cryptographically verifiable. Debugging = reading the reel. No hidden time-state.
- **No races, no async-hell.** Order is causal (the `p` links), not wall-clock; concurrency is
  parallel chains (branches) reconciled by convergence (the World = agreement among folds). Locks /
  data races / "happens-before" are _the_ time-logic that disappears.
- **Code and LLM-cognition unify.** If cognition emits Word one fact at a time, an LLM "running a
  program" = laying facts on a chain = exactly what code-cognition does. Two organs, one stream
  (llm.md) — because _the loop is the same loop_ (grow the chain).
- **It's the convergence of four models** that today are separate patterns: event-sourcing (facts =
  the log, fold = the projection), immutable/functional (every change a new value), reactive
  dataflow (see-as-noun: cells recompute from the fold), and logic/relational (declare + query,
  search.md). Chain-native programming is all four at once, with crypto-provenance + presentism.

## The honest seam (where it stays hard)

- **The bottom turtle is real.** You can't make _everything_ chain-native — at some floor a real CPU
  runs real instructions in real time (`host:` = the bottom turtle, word.word doctrine). A
  fact-stamp is a hash + a persist; a JS for-loop is nanoseconds. So chain-native is for the
  **semantic** layer (the meaningful steps), not inner-loop arithmetic. The art is pushing the
  time-logic _down_ to the smallest irreducible floor and making everything above it timeless.
- **Cost tracks surprise, not size** — the mitigation is delta/predictive-coding (llm.md): a
  near-identical step folds the _difference_. Honest caveat: compute can be delta-cheap but **the
  stamp stays whole** (a hash chain can't take a delta and stay self-contained). So the chain-native
  layer is "one stamp per _meaningful_ step," with the host floor doing the cheap inner work.

## The one line

Today the Word is timeless but rides a time-based engine. Chain-native programming makes the
_engine_ timeless too: **loops become folds, state becomes the fold, control flow becomes the
conditional Word, the program counter becomes the chain head** — and the only time left is the
irreducible host floor at the very bottom. Programming, rebuilt on causality instead of the clock.
The core move is one substitution: replace the clock with the chain. Ordinary programs run on time — a counter that advances, memory that mutates step by step, execution ordered by when things happen. Chain-native programming throws all three out and runs on causal order: the chain head is the program counter, the fold is the memory, and "what happens next" is just "what fact gets laid next." Nothing is ordered by a clock; everything is ordered by what it chains off.
Once you make that swap, every classic construct has a forced translation. A loop is a fold that grows the chain — each iteration lays a fact off the last, the head advances, the loop ends when a condition read from the fold goes true. State is never a variable; it's the fold of the facts so far, so the value at any point is exactly what the chain reduces to there. No mutation, so no stale reads, no aliasing. Control flow is the conditional Word, and a branch is a history fork. A whole program is a Book: content-addressed, forkable, replayable, verifiable, pausable — a paused program is just a chain head, resume is laying the next fact.
This is the is-be blade again, one level up from 17.md. There you put it on the wire: one act, the stamp, with see as the inert read. Here it's on the loop — each iteration is a see (read the fold, inert) then a stamp (lay the fact). A chain-native loop is a string of is-be steps. 17.md was the protocol edition; this is the control-flow edition, the same primitive propagating down the stack.
Now the honest situating, because "rebuilding programming from the ground up" is true but not from nothing. Each piece exists somewhere, each having killed one part of clock-time already: event sourcing made state a log of facts you replay; functional/immutable made every change a new value; Datomic made state the fold of immutable facts with time as data instead of a clock (that's your "state is the fold, never a variable" almost verbatim — the closest existing sibling); git and CRDTs made concurrency branch-and-converge instead of locks and happens-before; dataflow made cells recompute from inputs (your see-as-noun); logic programming made computation declare-and-query. Your move isn't inventing one of these. It's running all of them on one substrate at once, which none of them do — and that's real, not a relabel.
What's new on top is two things. The crypto-provenance plus presentism: every run is a signed, content-addressed, verifiable fact-chain, and a being is the head, not a process in memory. And the part none of those systems reach — cognition and code on one primitive. Datomic doesn't think; an LLM doesn't replay-audit. But if a mind emits Word one fact at a time and a program runs by laying facts one at a time, they are the same loop, and a thought and a computation become indistinguishable at the substrate. That's the payoff the whole conversation has circled: the stamp is the only act, whether the thing stamping is reasoning or executing.
The one bound — and the doc names it — is that you don't really remove time, you relocate it. A real CPU runs real instructions in real nanoseconds at the floor. So chain-native is the semantic layer: one stamp per meaningful step, a thin host floor doing the cheap inner arithmetic in ordinary time. "Remove time" means "confine time to the smallest irreducible floor and make everything above it causal." Built-out-of: yes, most of Word can be built out of Word. That's self-hosting, the maximal form of your existing rule, "compositions over code except for genuine primitives." Push it all the way and the parser, the evaluator above the floor, the now-collapsed dispatch, the standard library, control flow as conditional Words — all become .word, facts in the fold. What can't become Word is a small kernel: the crypto (Blake3), the content-addressed store, raw IO, and the minimal evaluator that reads a .word and applies a word-definition. You can't hash with a thing that needs hashing to run; the floor stays host. So the endpoint isn't "no coding background," it's "no JS above the kernel" — Word down to a tiny seed.
And the base can be Word describing Word — word.word defines word, verb defines verb. That's a metacircular kernel, real, with one classic catch: chicken-and-egg at ignition. Before any Word-evaluator exists in Word, something has to read the first foundation .words and build the initial fold. That's a one-time bootstrap seed in the host. After ignition, Word runs Word. Every self-hosted language does exactly this: write a minimal version in another language, rewrite the language in itself, then let it build itself forever after.
Now "run JavaScript and other code" — being chain-native gives you nothing here for free. Foreign code carries foreign semantics: mutable state, wall-clock order, the precise time-logic Word exists to escape. So running JS leaves you two doors, both keeping a bridge. Either you embed a real JS engine and call across a boundary (a foreign capability at the edge, your host: / adapter shape), or you transpile JS into Word — at which point it isn't JS anymore, it's Word wearing a JS source file, running chain-native with JS's real semantics thrown away. There's no third door where Word natively is a JS runtime. The bridge is the membrane between Word and not-Word, and you want it thin and explicit, not dissolved — absorbing JS's mutable-time semantics would drag back the exact mismatch doc 20 removes.
So the clean target: Word built out of Word down to a kernel (implementation dependency gone), with explicit bridges at the edges to whatever foreign code or devices you need to use (interop dependency, unavoidable, correctly fenced). Self-hosting kills the first. Nothing kills the second, because the second just is "Word touching the world that isn't Word."
The gain is large: the engine becomes a Book — content-addressed, forkable, verifiable in its own terms. You can verify the compiler the way you verify any reel, and doc 20's mismatch (a timeless surface on a time-based engine) dissolves everywhere except the kernel floor. That floor is doc 20's "push time to the smallest place," achieved.
One caveat, performance. A Word evaluator written in Word, interpreting Word interpreting Word, is a tower of interpreters and it crawls. Real self-hosting escapes by compiling, not interpreting all the way up: the kernel compiles Word down to fast kernel-ops (bytecode, WASM, native), and the tower collapses to one fast floor. So what you self-host is a Word compiler, and the kernel does the cheap inner work — the same split as doc 20's "one stamp per meaningful step, host floor does the arithmetic." Interpret Word-in-Word forever and it's beautiful and unusably slow. Compile it and it's beautiful and real.

Benefits, the ones specific to your system rather than generic self-hosting:

The engine becomes verifiable. Today the implementation is a trusted black box: you run the program, you trust the JS underneath. If the engine is .word facts, signed and content-addressed, you verify it the way you verify any reel. You can prove which engine ran a program and who authored each word of it. The last black box, the substrate, closes.
Determinism, which is what makes that verification mean anything. JS leaks nondeterminism into semantics: JIT, GC pauses, float quirks, platform differences. Shrink it to a small specified kernel and the same kernel plus the same engine-Book plus the same input gives a byte-identical result. Replay-verification only works because execution is deterministic; self-hosting plugs the leak JS opens.
The language versions and forks like data. A new engine is a new content-addressed Book. Programs pinned to an engine-hash run identically forever, two engine versions coexist with zero conflict, and "which Node / works on my machine" disappears. A lockfile for the language itself.
Extending the language is just stamping a Word. New primitives, control forms, vocabulary become signed declarations on the chain — auditable, hot-loadable, grown by accretion instead of privileged compiler edits. And beings can extend their own language, which is the emergent-language experiment going native.
Sovereignty and portability. The foreign dependency shrinks from the whole V8/Node/npm tower to a tiny kernel you control; everything above is sealed and yours. Porting Word to a new machine becomes porting the kernel, not the engine, and the same Word runs on any kernel.
The bound: none of this reaches the kernel itself. It stays the trusted floor, which is why it has to be tiny and hard-audited — the one thing you still trust the old way. And all of it is gated on the next thing.

What compiling Word is. Right now the Word is interpreted: every run, the evaluator re-reads the AST, re-resolves each word against the fold, re-dispatches, stamps. It decides everything fresh, every time. Compiling does that deciding once, ahead of time, and bakes the result into a flat sequence of the kernel's fast primitive ops — hash, persist, fold-step, compare, branch. You resolve the word against the fold once, expand its compositions, lower its control flow (the see-reads and stamps, the conditional Words, the loops-as-folds) into a fixed instruction list, and emit that. Running then skips the parsing and resolving and just executes the precomputed steps.

Two things make it yours, not generic. The inlining is your compression idea at the machine level: a deep word built from a thousand sub-words flattens into one compiled routine, the same way a recurring mark crystallizes into a single sign. And the compiled output is itself a content-addressed, signed Book — "this source-Book, through this compiler-Book, yields this compiled-Book," a fact like any other. So compilation is verifiable and reproducible (same source plus same compiler equals the same hash), and content-addressing caches it for free: a Word's compiled form is pinned by hash, so nothing recompiles the same Word twice, anywhere.

And it's what makes self-hosting fast. The compiler, written in Word, runs once to translate Word into kernel-ops; the kernel runs those ops fast forever after. The tower-of-interpreters slowness gets confined to compile time, which happens once. You think in the slow high language and run in the fast low one — every self-hosted language's trick, and yours carries crypto-provenance on both ends.
