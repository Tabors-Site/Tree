# The Word: the IR (Phase 1, the host-agnostic contract)

The clause-grained AST that a `.word` program parses into and the evaluator runs. The surface (prose) can evolve; this IR is the stable, content-addressed, versioned contract. A fact's content IS an IR act node, one schema (see [1.md](./1.md), "The Word and the fact-chain"). Built before the surface grammar (2.md, Phase 1). This is v0, the contract to iterate against the gate; field names are provisional.

## The atom: the clause

Every node is a clause: one subject, one verb, one operation. Plain JSON, an explicit `kind`, deterministic field order.

```
{ kind, verb, op, by, through, of, to, at, on, when, host }   // not every field on every node
```

- `verb` — the operation. Grounds out in one of the five: `see`, `do`, `summon`, `be`, `name`, or a defined word (rule 8) whose body grounds out in them.
- `op` — the specific operation within that verb (`do` + `create-space`, `be` + `birth`), the registry name. Surfaced building the cherub slice (Phase 2).
- `by` — the actor: always a Name reference (rules 9, 11), the signer, `I` / `Name` / `I_AM`; never a key value, and never a being.
- `through` — the being the Name acts through (the being, the body with position and can-do). Present for world acts; absent for NAME-layer acts (mint, declare, banish), where the Name acts as itself. The parser fills both `by` and `through` from context, whether the surface led with the Name (`I queue`) or the being (`Tabor queues`).
- `of` / `to` / `at` / `on` — the objects: what is acted on, the receiver (rule 17), the place, the address.
- Surface word order (subject-first or verb-first) is presentation; both parse to the same node.

## Node kinds

Three authored kinds, plus the parts they are built from. Facts are NOT a kind: a fact is an act node sealed and wrapped in the host chain-envelope.

- **declaration** (the law, standing):
  - `kind: "is"` — structure or a kind: `A music room is a space`, `A drum is matter`, `a dj is a role for a music room` (rule 8). Fields: `subject`, `isA`, `scope`.
  - `kind: "can"` — a capability on a role (rule 8): `a dj can queue a song`. Fields: `role`, `verb`, `of`.
  - `kind: "cannot"` — a prohibition (rule 14): subtracts a capability and outranks a `can` on the same act. Fields: `subject`, `verb`, `of`, `exception`.
  - `kind: "flow"` — a standing watch (rule 6): `when X, do Y`. Fields: `condition`, `effects`. Dormant, no chain entry until it fires.
- **act** (the life, a deed performed; sealed, it is a fact). Fields: `verb` (one of five), `op` (the operation within the verb), `by` (Name ref, the signer), `through` (being ref, the being; absent for NAME-layer acts), `of` / `to` / `at` / `on`, `effects?`. An act may carry a receiver (`to`, rule 17), a bridge address (`on: here :: there`, rule 15), or a position move (rule 18). `effects` are nested act nodes for the distinct further facts it causes; absent for a self-contained act (whose only fact is itself).
- **closure** (rule 13): names a bounded span: `this was the first day`. Fields: `name`, `closes` (the span boundary).
- **parts** (inside the above):
  - **reference** — to a being (Capital, bare), a kind (lowercase, with an article, which flips the reference between the kind and an instance), a Name reference (`I` / `Name` / `I_AM`), an address (`reality#branch/space`, rule 15), a named span, or a matter / space.
  - **condition** — for a flow: a pattern over the world (a being's act, a state, a clock being's beat). Relational (rule 4); never elapsed wall-time.
  - **compute-call** — the escape hatch (`host: builtin`): names a host builtin, args in, result out. The only opaque node, and it stays narrow: it qualifies only for genuine external computation (crypto, parsing, an ML model, network IO), never as a shortcut for substrate-shaped logic, which stays native IR even when verbose. If it becomes the default for anything tricky, the IR's content-addressing, determinism, and federability erode. Deterministic builtins are specified by vectors; effectful compute stays outside the replayable core.

## Registers (read off the verb form, not stored as a field)

- **LAW** = declarations and flows (modal or general present: `can`, `when`, `is`).
- **LIFE** = acts, sealed (past). A fact = an act node + the host chain-envelope (hash, prev, sig, seq, branch), folded onto every reel it touches (one fact, not duplicated).
- **NOW** = the fold (state), derived from the life; not an authored node.

## Where the 18 rules land

- **1** active: an act's `by` is the actor; nothing passive or self-as-object hides it.
- **2** possession: `contains` and `owns` are the only possession relations (state, not verbs).
- **3** no host-mechanism: no `stamp` / `fold` nodes; acts auto-seal.
- **4 / 12 / 13** time: conditions are relational; a clock is a being whose beats are act nodes; closures name spans; there is no elapsed-wall-time node.
- **5 / 9 / 10 / 11** identity: `by` is the Name (the signer, `I` / `Name` / `I_AM`), `through` is the being it acts through (the being); a NAME-layer act carries `by` and no `through`. Beings are Capital and bare, kinds lowercase with an article; no key value appears anywhere.
- **6** conditionals: the `flow` node, dormant until it fires.
- **7** person: `by` resolves `I` / `you` to the acting or addressed Name; one node, two render faces.
- **8** roles and composition: `is` declares kinds (being-kinds are roles, plus matter-kinds and space-kinds); a defined act is a kind whose body is acts.
- **14** prohibition: the `cannot` node, with precedence over `can`.
- **15** bridge: an act's address is a two-stance `here :: there`, left full, right possibly bare.
- **17** receiver: the `to` field on an act.
- **18** position: a being's `in` relation; movement is a `do` act, never a be op.

## Contract properties

- **Content-addressable:** a compiled node has a stable id (its hash) and federates like any resource.
- **Versioned:** a schema version, so grammar changes are explicit and the Phase 6 conformance suite detects drift.
- **Deterministic:** fixed field order, fixed number handling, no wall-clock or randomness inside evaluation (those enter only as signed inputs or a clock being's beats). Required by CAS replay today, not just by a future host (the 2.md invariant).

## Conformance vectors to pin early

The error-prone compositions. Freeze these the way the canon vectors are frozen (Phase 6), but write the first ones now so the parser, evaluator, and renderer are built to agree:

- **Prohibition precedence** (rule 14): a role with both `a dj can queue a song` and `a dj cannot queue while the room is closed`. When the room is closed the `cannot` wins, and all three (parse, evaluate, render) must compose it identically. Keep the condition relational (a state); a weekend or an hour needs a clock being, not wall-time.
- **NAME-layer versus world act** (the `by` / `through` split): `mint a new Name` carries `by` and no `through`; `Tabor queues a song` carries both. The evaluator must never assume a being.
- **Escape-hatch determinism**: a compute-call result feeds back so that two replays produce identical facts.

## Gate (2.md, Phase 1)

The IR can represent the hardest existing slice with no opaque host escape beyond named compute-calls. Candidate slice for Phase 2 (evaluator first): **the cherub register flow** is the stronger first choice. It has the most distinct mechanisms at once, the BE lifecycle, the escape hatch (`host: keys.generate`), a NAME-layer mint (the `by`-without-`through` case above), session binding, and the first-birth rule, so it exercises exactly the IR features most likely to be wrong. Harmony (the drummer plus dancers) is the natural second; it stresses a different bug class, the pulse and beat-driven coordination (rule 12, the self-rescheduling recurrence via the wake system). Pick the harder one first; the easier one will work and reveal less.
