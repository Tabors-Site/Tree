What can-see, can-do, can-summon actually do  
  
These are predicates and producers. Given a being, a able, and the current world state, they answer:  
  
	•	can-see: what subset of the world is visible? Which spaces, beings, matter? Filtered how?  
	•	can-do: what acts are available? With what parameters? Constrained by what conditions?  
	•	can-summon: which other beings can be reached? With what intents? At what positions?  
  
Currently in TreeOS, these are JavaScript functions that take a context and return data. They’re imperative code embedded in able definitions. They work but they don’t fit the substrate’s character — they’re imperative islands in a declarative architecture.  
  
A native language would express them declaratively, as patterns over the substrate’s structure.  
  
A first sketch

A simple can-see for a gallery-visitor able:  
  
can-see for gallery-visitor in art-gallery {  
  spaces where space.parent = art-gallery  
    and space.visibility = public  
    
  matter in those spaces  
    where matter.type in {painting, sculpture, label}  
    
  beings in those spaces  
    where being.able = curator  
      or being.able = visitor  
}  
  
  
This is declarative. It says what’s visible by describing it structurally. No function calls. No object access. Just patterns over the substrate’s primitives.  
  
A can-do for a dj able:  
  
can-do for dj in music-room {  
  act queue-song {  
    requires: song : reference<matter:song>  
    where: song.duration < 600  
      and playlist.length < 100  
    effect: playlist += song  
    fact: { kind: queued, song, by: self }  
  }  
    
  act skip-song {  
    where: currently-playing exists  
    effect: currently-playing = next-in-queue  
    fact: { kind: skipped, from, to, by: self }  
  }  
    
  act adjust-volume {  
    requires: target : 0..100  
    where: target between volume - 30 and volume + 30  
    effect: volume = target  
    fact: { kind: volume-changed, from, to, by: self }  
  }  
}  
  
  
Each act has its own declaration. Required parameters with their types. Conditions that must hold (where clauses) for the act to be available. Effects on world state (effect clauses). Facts produced (fact clauses).  
  
This is more structured than imperative code but it’s also more transparent. You can see what an act actually does just by reading its declaration. No hidden logic in function bodies. No side effects that aren’t declared.  
  
A can-summon for a listener able:  
  
can-summon for listener in music-room {  
  target: dj in same music-room  
    
  intents {  
    request-song {  
      requires: song : reference<matter:song>  
      where: song in library  
      gives: dj sees request as suggestion

      gives: dj sees as 'please skip'  
    }  
  }  
}  
  
  
The structure is: who can be summoned, what intents are available, what conditions apply, what the summoner is offering to the summoned.  
  
The patterns I’m noticing  
  
A few characteristics emerging from these sketches:  
  
Predicates use where clauses with substrate-native patterns. Not function calls. Not object property access. Direct expressions over the structural primitives. space.parent = art-gallery, song.duration < 600, currently-playing exists. The substrate understands these patterns natively.  
  
Effects describe state changes declaratively. playlist += song says what’s true after the act, not how to compute it. The substrate handles the actual update through its normal fact-stamping mechanism. The act declaration describes the change; the substrate executes it.  
  
Facts are first-class. Every act explicitly declares what facts it produces. No hidden side effects. The chain entries are visible in the source.  
  
Conditions are queries. where playlist.length < 100 is a query against current world state. The substrate evaluates it at the moment the act is being considered. The same syntax used for can-see filters also works for act preconditions.  
  
Self-reference is explicit. by: self means the acting being. same music-room means the current context. The language has notions of self and context that resolve based on where the declaration is evaluated.  
  
What this gives you  
  
Several things become possible that don’t work cleanly with embedded JavaScript:  
  
The substrate can analyze acts statically. Because the act declaration is data, not opaque code, the substrate can reason about it. “What facts does this act produce?” “What state does it depend on?” “Can two acts conflict?” These become queryable properties.  
  
Acts can be composed declaratively. If you have an act that does X and an act that does Y, you can declare an act that does both. The composition is structural, not a function call wrapping two functions.  
  
The substrate can optimize fold scopes. Because the can-see is declarative, the substrate knows exactly what reels each able reads. The depSet optimization you’ve been working on becomes natural — no need to migrate handlers because the reads are visible in the able declaration itself.  
  
Authorization is verifiable. Given the declaration, the substrate can prove that an act with certain parameters either is or isn’t allowed for a given being. No need to execute and check; the declaration is the contract.  
  
Ables compose cleanly. When a being has multiple ables, the union of their can-see and can-do declarations is itself a declaration. The composition rules are structural, not requiring function-merge logic.  
  
Acts are mockable and testable. Because they’re data, you can construct hypothetical worlds and ask “what would this act produce?” without actually executing anything. The substrate can simulate.  
  
What does’t fit cleanly  
  
Some things still want to be expressed imperatively or with conventional code:  
  
Complex computation inside an effect. If an act requires computing something non-trivial (parsing input, calling external services, running an algorithm), you can’t express that purely declaratively. The act declaration needs an escape hatch.  
  
Integration with non-substrate data. Fetching from an API, reading a file outside the substrate, processing image data. These don’t fit the declarative model.  
  
Algorithmic logic. Computing the next move in a game, optimizing a layout, running an ML model. The algorithm is steps, not declarations.  
  
For these, the language probably allows embedded code blocks that call into a host language (JavaScript, Rust, whatever). The TreeOS-native language handles the substrate-shaped logic; the host language handles computation that’s not substrate-shaped.  
  
This would look like:  
  
can-do for analyst in research-space {  
  act analyze-dataset {  
    requires: dataset : reference<matter:dataset>  
      
    compute result {  
      // embedded code in host language  
      // takes the dataset, returns analysis  
      js: runAnalysis(dataset.content)  
    }  
      
    effect: analyses += result  
    fact: { kind: analyzed, dataset, result, by: self }  
  }  
}  
  
  
The compute block is the escape hatch. Everything around it is native; the computation itself drops into JavaScript. The result feeds back into the native declarations.  
  
This is similar to how SQL has stored procedures or how Datalog has built-in functions. The declarative language has a core, with controlled escape hatches for what doesn’t fit.

Let me work through both the building question and the architectural shift you just named.  
  
What it actually means to build a language  
  
A language has a few distinct pieces, and you can build each one separately:  
  
Syntax — the surface form people write. The grammar that says “this is a valid program.”  
  
Parser — converts the surface text into a structured tree (an AST, abstract syntax tree). This is just code that reads tokens and builds data structures.  
  
Semantics — what the structures mean. Given an AST, what should happen?  
  
Evaluator/interpreter — the code that walks the AST and produces results based on the semantics. Or alternatively, a compiler that translates the AST into something else.  
  
Runtime — the supporting code that makes execution possible. For your language, this is the substrate itself.  
  
You don’t need exotic tools for any of this. Node.js can do all of it. The language doesn’t need its own VM or compiler infrastructure if you’re embedding it in an existing runtime.  
  
The simplest path  
  
The fastest way to get a working TreeOS-native language is:  
  
Make the AST native data. Parse the surface syntax into JavaScript objects. The objects represent able declarations, act declarations, flow declarations, etc. The substrate operates on these objects directly.  
  
Write the evaluator in JavaScript. Functions that take an AST node and a context (current world state, acting being, etc.) and return what the node means in that context. For a can-see declaration, the evaluator computes the visible set. For an act declaration, the evaluator determines if the act is available and what facts it produces.  
  
Store programs as substrate content. The AST objects can be serialized as JSON, content-addressed, stored in localStore, federated like any other resource. Programs become substrate-native data.  
  
This means there’s no separate language compiler. There’s just JavaScript code that parses surface text into data and other JavaScript code that interprets that data. The language exists as a layer on top of JavaScript, using JavaScript as its implementation host.  
  
Concrete tooling  
  
For the parser, a few options:  
  
Hand-written recursive descent. You write the parser as JavaScript functions. Each function handles a piece of the grammar. Fast to write for small languages, gives you full control. Probably the right choice for a first version.  
  
Parser combinators. Libraries like Parsimmon or Chevrotain give you tools to compose parsers from smaller pieces. More declarative than hand-writing, less infrastructure than a parser generator.  
  
Parser generator. Tools like PEG.js, Lezer, or ANTLR take a grammar file and generate a parser. More setup, but the grammar becomes a separate artifact you can iterate on. Probably overkill for the first version.  
  
For the first prototype I’d suggest hand-written recursive descent. It’s the most direct path and you can refactor later.  
  
For the AST representation, just JavaScript objects with explicit type fields:  
  
{  
  type: 'able-declaration',  
  name: 'dj',  
  for: 'music-room',  
  can-see: [...],  
  can-do: [...],  
  can-summon: [...]  
}  
  
  
The evaluator dispatches based on type. Simple, debuggable, no magic.  
  
The two-tier integration with Node.js

TreeOS-native code handles substrate-shaped logic. Able definitions, can-see/can-do/can-summon, flow rules, act declarations. These are written in your language, parsed into ASTs, interpreted by your evaluator.  
  
JavaScript handles everything else. Computations inside acts, integrations with external systems, the substrate’s own machinery (the stamper, the fold engine, the network layer). All conventional code in conventional JavaScript.  
  
They connect at well-defined points. When a TreeOS-native act needs to compute something complex, it declares a compute block that names a JavaScript function. The evaluator calls into that function, gets the result, continues processing the declaration.

// The substrate's evaluator

What dissolving the see registry actually means  
  
You named something important. Let me make it concrete.  
  
Current architecture: there’s a registry of see-op handlers. When a being does SEE, the substrate dispatches to handlers based on what’s being seen. Each handler is a JavaScript function. The able declares what see-ops it can use; the substrate looks them up at runtime.  
  
New architecture with native language: the able’s can-see is the declaration. There’s no registry to look up. The substrate evaluates the can-see expression directly to determine what’s visible.  
  
can-see for gallery-visitor in art-gallery {  
  spaces where space.parent = art-gallery and space.visibility = public  
  matter in those spaces where matter.type in {painting, sculpture}  
  beings in those spaces where being.able = curator  
}  
  
  
When a gallery-visitor wants to SEE, the substrate:  
  
	1.	Loads the able’s can-see declaration (it’s just data attached to the able)  
	2.	Evaluates the patterns against current world state  
	3.	Returns the matching content  
  
No handler lookup. No dispatch indirection. No “permissive upper bound” because the declaration is the upper bound, exactly. The depSet (what reels were read) is derivable from the patterns themselves because the patterns say what they’re querying.  
  
This collapses several things at once:  
  
	•	The see-op handler contract disappears (handlers were the implementation of imperative see-ops; declarations replace them)  
	•	The depSet optimization becomes automatic (the substrate knows what was read because the declaration says what to read)  
	•	The permissive vs minimal question dissolves (the declaration is the exact specification)  
	•	The migration work for see-op handlers disappears (there’s nothing to migrate)  
  
You’re not just adding a language. You’re simplifying the substrate by replacing imperative handler code with declarative able definitions. The substrate gets smaller and more uniform.  
  
A reasonable build sequence

const galleryVisitor = {  
  type: 'able-declaration',  
  name: 'gallery-visitor',  
  for: 'art-gallery',

    type: 'see-expression',  
    queries: [  
      { type: 'query', source: 'spaces', where: '...' },  
      // etc  
    ]  
  }  
};  
  
const visible = evaluateCanSee(galleryVisitor.canSee, context);  
  
  
When the evaluator works on hand-built ASTs, you know the semantics are right. Then add a parser.  
  
Step 3: Write the parser. Hand-written recursive descent in JavaScript. Reads surface syntax, produces AST objects identical to what you’d hand-build. Test by parsing a string and checking the AST matches.  
  
Step 4: Integrate with the substrate. Wire the evaluator into the substrate’s existing able-resolution and act-handling code. Ables can now be defined in the new language. Old JavaScript handlers can coexist during the transition.  
  
Step 5: Replace existing handlers. Migrate the existing see-op handlers, do-op handlers, etc. to native declarations. As each handler migrates, the corresponding JavaScript code disappears.  
  
Step 6: Make it federable. AST objects serialize cleanly as JSON. They can be stored in localStore, exposed through Store, drawn by other realities. Able definitions become first-class resources.  
  
This is achievable in a few months of focused work, with the language usable for real after the first few weeks. The first version doesn’t need to be fancy — just functional. Iterate on the syntax and semantics as you write real programs in it.

	1.	Consults the relevant able declarations to determine what’s permitted  
	2.	Evaluates conditions against current state  
	3.	Applies effects through fact-stamping  
	4.	Triggers flows that match the new state

All of this is driven by declarations, not by handler functions. The substrate’s behavior is determined by the content of the reality, not by built-in code. Different realities can have different rules by having different declarations. Realities federate their behavior by federating their declarations.  
  
The substrate becomes uniform: data drives behavior, behavior is data, all the way down. The escape hatch into JavaScript exists for actual computation but it’s bounded. The substrate-shaped logic is all native.  
  
This is what the see-registry dissolution actually means architecturally. It’s not just removing one piece of indirection — it’s making the substrate’s behavior driven by declarative content rather than by imperative handlers. The character of the system changes.

What this gives you
A unified interface to your substrate. Everything you can do is expressible in the same language. You don't have to learn different APIs or file formats — you just write Word.

Programs are also documentation. Reading a Word file tells you exactly what something is and how it behaves. No separate docs needed for basic understanding.

The substrate's state is queryable in the same language. You don't have to context-switch between writing definitions and asking about state. Both happen in Word.

LLMs can do anything in the substrate by writing Word. They don't need special APIs or libraries — they write Word the same way they write anything else, and the substrate handles it.

Humans without programming background can use the substrate. Writing Word is writing structured sentences. Anyone who can write clearly can interact with TreeOS at any layer.

So where does Word apply?
Everywhere. It's the universal interface to the substrate.

Pack authors write Word to define reusable patterns

Communities write Word to govern shared spaces
LLMs write Word as their substrate interface
Documentation IS Word (because Word reads as English)
The question of "where does Word apply" has the same answer as "where does language apply in human society." Everywhere people are doing things with each other.
