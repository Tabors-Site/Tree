# 15 ‚Äî materials/ symmetry audit

The kinds ‚Äî being, matter, space, name, history ‚Äî are the same kind of thing: a noun the factory folds from facts, so their folders should read as one shape repeated. The good news, after reading the code rather than listing filenames: they are **more symmetric than they look**, and the Name/Being split is **more done than it looks**. This audit marks the shared shape, each kind's own nature, and the one narrow thing genuinely out of place. publish/ is out of scope (it becomes the book system).

## The shared shape

A kind folder is the **host** for that kind's **word**. The core every kind shares:

- `<kind>.js` ‚Äî the schema (the kind's `is`).
- `reducer.js` ‚Äî the fold, facts ‚Üí state.

That is the symmetric heart, and **all five kinds have both, same role each.** Everything past it is the kind's do-ops, its read surface, or its own nature ‚Äî and those legitimately differ.

## Two things I first read as asymmetry ‚Äî and they're not

**The "five catalog names" are six different roles.** `beingsCatalog` = a global *list of being-presences* (homeSpace, position, cognition); `matters.js` = the matter core + CRUD; `spaces.js` = space *operations*; `name/registry.js` = a name-token ‚Üí pubkey *resolver*; `histories.js` = read helpers; `historiesCatalog.js` = the branch-graph SEE catalog. Different jobs, each named for what it does ‚Äî forcing one `<kind>s.js` convention would *mis*-name them. Leave them.

**The "missing ops.js" in name/history is correct.** name has `registerOperation` nowhere ‚Äî it acts at the **verb** layer (NAME: declare / banish), not via do-ops; history's changes ride BE + `branchCreation`. Not every kind has do-ops.

(Minor, real, just write it down: `<kind>Id.js` exists only where the id is *computed* ‚Äî being, matter ‚Äî and not where it is given: space uuid, name pubkey, history string. State the rule so the absence reads as intentional.)

## The Name/Being split ‚Äî mostly done, one narrow residue

The split lifted the Name out of the Being. Reading being/identity/ carefully, **most of it is correctly placed**, and my first-pass "move it all to name/" was wrong:

| file | what it actually is | verdict |
| --- | --- | --- |
| `lineage.js` | the **being-tree** ‚Äî who performed the birth act; credential authority over a being's tree position | **stays being/** |
| `inheritation.js` | the **being-tree** authority axis (`hasAuthorityOver` over positions) | **stays being/** |
| `lookups.js` | being read-side (findBeingByName, isFirstBeing, ‚Ä¶) | **stays being/** |
| `identity.js` | a thin **re-export aggregator** over lookups/credentials/birth | stays (re-points on split) |
| `credentials.js` | the **auth / session layer** ‚Äî scrypt passwords + session JWTs + at-rest encryption. Cross-cutting *by nature*: a Name auths through a Being, the token carries both `beingId` and `nameId`, and the crypto is generic (the being's password *and* the name's key both run through `encryptCredential`) | **stays** ‚Äî shared auth, not name-stuff |
| `birth.js` | **mixed** ‚Äî mints the keypair (Name) + forms the body (Being) | **split** |

So the genuine "name stuff in being" is narrower still: **just the keypair-minting inside `birth.js`** (`birthBeing` mints the Name's keypair while forming the body). `credentials.js` looked like a target on the filename but is the cross-cutting auth layer ‚Äî its crypto serves both the being's password and the name's key, so it is shared, not misfiled. The being-tree, the lookups, and the auth layer are all correctly where they are. "No name stuff in being" is *essentially already true* ‚Äî bar that one mint.

One small leak the other way: `name/closure.js` imports `I_AM` from `being/seedBeings.js`. I_AM is a *Name* (name.js: `_id="i-am"`), so its constant wants a name/ home.

## Done this pass

The split closed faster than the audit feared:

- **Minting ‚Üí name/ (done).** The credential crypto/minting (`encryptCredential` / `decryptCredential` / `hashPassword` / `comparePassword` / `mintCredentialSpec`) moved to **name/credentials.js**. being/identity/credentials.js is now the session/token layer (cross-cutting auth) that imports + re-exports the crypto, so nothing broke (verify-credreset-cut 7/7). And the keypair-mint I'd flagged in `birth.js` turned out to **already** live in `nameOps.declareHandler` ("key-minting LIVES now ‚Äî it left birth.js"), so there was nothing to move there. All minting is the Name's now; being/ is clean of it. birthBeing forms the body + references a Name + optionally mints a credential (via name/credentials).
- **Added: `set-password`** ‚Äî a Name adds/changes/removes its password after declare, proving it holds the key (importKey, an unlocked session, or system-decrypt when no password yet); the new `privateKeyEnc` rides the fact as a censored `SECRET_KEY`.

**I_AM's constant ‚Äî resolved, a non-issue.** seedBeings.js says it plainly: `I_AM = "i-am"` is "the first being's name AND id" ‚Äî the I-Am Being row's primary key *and* the root Name's id (name.js: `_id="i-am"`). It is the **dual root**, one shared constant: a being constant (the root Being's pk) that doubles as the Name id. So name/closure importing it is the *shared root*, not misplaced name-stuff. Moving it is high-churn (imported everywhere) for an ambiguous gain ‚Äî it stays.

**Conventions ‚Äî documented (here).** The id rule: `<kind>Id.js` only where the id is *computed* (being content-id, matter id), not where it's given (space uuid, name pubkey, history string). And name/history have no `ops.js` because their ops are **verb-level** (NAME's declare/connect/release/set-password/banish; history's branch acts ride BE). This audit is the written-down rule, so the absences read as intentional.

## Noted for after: the NAME ops should be `.word`

The NAME ops (declare / connect / release / set-password / banish) are still a frozen JS **Map** ‚Äî `NAME_OPS` in nameOps.js ‚Äî a **silent register**: undeclared on the chain, not a word. That is precisely the shape the do-op cutover removed (the five Maps ‚Üí the fold of `declare-word` facts). By the doctrine ‚Äî everything is a word, and the system should only run words it knows from a declared fact ‚Äî the NAME verb's ops belong in the fold too: declared and folded like the do-ops, not held in JS. A follow-on to the word migration (alongside roleWordRegistry ‚Üí wordStore): cut `NAME_OPS` over to declared words. `set-password` lands in JS for now, but it is the next thing that should become a word ‚Äî and the general rule is that any op living only in a JS Map, never declared, is a silent register to fold.
