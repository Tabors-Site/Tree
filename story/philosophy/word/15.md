# 15 — materials/ symmetry audit

The kinds — being, matter, space, name, history — are the same kind of thing: a noun the factory folds from facts, so their folders should read as one shape repeated. The good news, after reading the code rather than listing filenames: they are **more symmetric than they look**, and the Name/Being split is **more done than it looks**. This audit marks the shared shape, each kind's own nature, and the one narrow thing genuinely out of place. publish/ is out of scope (it becomes the book system).

## The shared shape

A kind folder is the **host** for that kind's **word**. The core every kind shares:

- `<kind>.js` — the schema (the kind's `is`).
- `reducer.js` — the fold, facts → state.

That is the symmetric heart, and **all five kinds have both, same role each.** Everything past it is the kind's do-ops, its read surface, or its own nature — and those legitimately differ.

## Two things I first read as asymmetry — and they're not

**The "five catalog names" are six different roles.** `beingsCatalog` = a global *list of being-presences* (homeSpace, position, cognition); `matters.js` = the matter core + CRUD; `spaces.js` = space *operations*; `name/registry.js` = a name-token → pubkey *resolver*; `histories.js` = read helpers; `historiesCatalog.js` = the branch-graph SEE catalog. Different jobs, each named for what it does — forcing one `<kind>s.js` convention would *mis*-name them. Leave them.

**The "missing ops.js" in name/history is correct.** name has `registerOperation` nowhere — it acts at the **verb** layer (NAME: declare / banish), not via do-ops; history's changes ride BE + `branchCreation`. Not every kind has do-ops.

(Minor, real, just write it down: `<kind>Id.js` exists only where the id is *computed* — being, matter — and not where it is given: space uuid, name pubkey, history string. State the rule so the absence reads as intentional.)

## The Name/Being split — mostly done, one narrow residue

The split lifted the Name out of the Being. Reading being/identity/ carefully, **most of it is correctly placed**, and my first-pass "move it all to name/" was wrong:

| file | what it actually is | verdict |
| --- | --- | --- |
| `lineage.js` | the **being-tree** — who performed the birth act; credential authority over a being's tree position | **stays being/** |
| `inheritation.js` | the **being-tree** authority axis (`hasAuthorityOver` over positions) | **stays being/** |
| `lookups.js` | being read-side (findBeingByName, isFirstBeing, …) | **stays being/** |
| `identity.js` | a thin **re-export aggregator** over lookups/credentials/birth | stays (re-points on split) |
| `credentials.js` | the **auth / session layer** — scrypt passwords + session JWTs + at-rest encryption. Cross-cutting *by nature*: a Name auths through a Being, the token carries both `beingId` and `nameId`, and the crypto is generic (the being's password *and* the name's key both run through `encryptCredential`) | **stays** — shared auth, not name-stuff |
| `birth.js` | **mixed** — mints the keypair (Name) + forms the body (Being) | **split** |

So the genuine "name stuff in being" is narrower still: **just the keypair-minting inside `birth.js`** (`birthBeing` mints the Name's keypair while forming the body). `credentials.js` looked like a target on the filename but is the cross-cutting auth layer — its crypto serves both the being's password and the name's key, so it is shared, not misfiled. The being-tree, the lookups, and the auth layer are all correctly where they are. "No name stuff in being" is *essentially already true* — bar that one mint.

One small leak the other way: `name/closure.js` imports `I_AM` from `being/seedBeings.js`. I_AM is a *Name* (name.js: `_id="i-am"`), so its constant wants a name/ home.

## Done this pass

The split closed faster than the audit feared:

- **Minting → name/ (done).** The credential crypto/minting (`encryptCredential` / `decryptCredential` / `hashPassword` / `comparePassword` / `mintCredentialSpec`) moved to **name/credentials.js**. being/identity/credentials.js is now the session/token layer (cross-cutting auth) that imports + re-exports the crypto, so nothing broke (verify-credreset-cut 7/7). And the keypair-mint I'd flagged in `birth.js` turned out to **already** live in `nameOps.declareHandler` ("key-minting LIVES now — it left birth.js"), so there was nothing to move there. All minting is the Name's now; being/ is clean of it. birthBeing forms the body + references a Name + optionally mints a credential (via name/credentials).
- **Added: `set-password`** — a Name adds/changes/removes its password after declare, proving it holds the key (importKey, an unlocked session, or system-decrypt when no password yet); the new `privateKeyEnc` rides the fact as a censored `SECRET_KEY`.

Left, minor: re-home I_AM's constant to name/; the small convention notes.

## Noted for after: the NAME ops should be `.word`

The NAME ops (declare / connect / release / set-password / banish) are still a frozen JS **Map** — `NAME_OPS` in nameOps.js — a **silent register**: undeclared on the chain, not a word. That is precisely the shape the do-op cutover removed (the five Maps → the fold of `declare-word` facts). By the doctrine — everything is a word, and the system should only run words it knows from a declared fact — the NAME verb's ops belong in the fold too: declared and folded like the do-ops, not held in JS. A follow-on to the word migration (alongside roleWordRegistry → wordStore): cut `NAME_OPS` over to declared words. `set-password` lands in JS for now, but it is the next thing that should become a word — and the general rule is that any op living only in a JS Map, never declared, is a silent register to fold.
