# 15 — materials/ symmetry audit

The kinds — being, matter, space, name, history — are the same kind of thing: a noun the factory folds from facts. So their folders should read as one shape repeated. They don't. This is the audit: the shape every kind should share, where the symmetry holds, where it's *accidentally* broken (clean it), where it's *inherently* different (keep it), and the one big structural gap — the Name/Being split is half-done. **publish/ is out of scope** — it becomes the book system; not touched here.

## The shape every kind shares

A kind folder is the **host** for that kind's **word** (being/ ↔ being.word: the folder implements what the word declares). Read against the words, each kind should have a parallel core:

| file | role | the word-face |
| --- | --- | --- |
| `<kind>.js` | the schema — the row shape | the kind's `is` |
| `reducer.js` | the fold — facts → state | how the word folds |
| `ops.js` | the do-op handlers — host for the kind's do-words | the kind's `do` |
| the catalog | CRUD / lookup over the projection | the kind's reads |
| `<kind>Id.js` | id derivation (only where the id is *computed*) | the kind's id rule |

Everything else in a kind folder is that kind's own nature (crypto for name, the tree for space, content for matter) — legitimately asymmetric.

## Where the core is symmetric — and where it's accidentally broken

| core | being | matter | space | name | history |
| --- | --- | --- | --- | --- | --- |
| `<kind>.js` schema | ✓ | ✓ | ✓ | ✓ | ✓ |
| `reducer.js` | ✓ | ✓ | ✓ | ✓ | ✓ |
| `ops.js` | ✓ | ✓ | ✓ | **✗** | **✗** |
| catalog | beingsCatalog | matters | spaces | registry | histories + historiesCatalog |
| `<kind>Id.js` | ✓ | ✓ | ✗ | ✗ | ✗ |
| `closure.js` | ✓ | ✗ | ✗ | ✓ | ✗ |

**Schema + reducer: clean.** All five kinds have `<kind>.js` and `reducer.js`, same role each. The symmetric heart holds.

**ops.js: a real gap.** being/matter/space have `ops.js` (the do-op handlers); name and history don't — their ops live unmarked elsewhere (name's verb-ops, history's `branchCreation.js`). Either give them an `ops.js` for parity, or document the exception in-folder. Right now you can't find a kind's ops by convention.

**The catalog: five names for one thing.** `beingsCatalog.js` / `matters.js` / `spaces.js` / `registry.js` / `histories.js`+`historiesCatalog.js` — same job (CRUD + lookup over the projection), five spellings. Pick one convention (`<kind>s.js` for the CRUD; `<kind>Catalog.js` only if a kind truly needs a separate read-catalog) and rename to it. history has BOTH `histories.js` and `historiesCatalog.js` — confirm that's two real roles, not a leftover.

**`<kind>Id.js`: half a convention.** being and matter have a dedicated id file; space, name, history don't (space id = opaque uuid, name id = the pubkey, history id = the branch string — nothing to derive). That's actually principled — a file only where the id is *computed* — but it's written down nowhere, so the absence reads as "missing" instead of "intentional." State the rule.

**`closure.js`: being and name only.** Both have it (the reel/lineage closure); matter/space/history don't. Confirm the other three genuinely don't need one.

## The big one: the Name/Being split is half-done

This is the "name stuff in being" — the largest asymmetry by far, and not a comment but the whole identity layer. The Name was lifted out of Being as a primitive; `name/name.js` documents it itself ("keys belong to identities, and the identity is the Name; it moved UP from `Being.qualities.auth.privateKeyEnc`"). But the **identity layer's code never moved.** It still lives under being/:

| file | lines | belongs to |
| --- | --- | --- |
| `being/identity/birth.js` | 865 | mixed — mints the keypair (Name) + forms the body (Being) |
| `being/identity/credentials.js` | 415 | Name (private-key encryption, custody) |
| `being/identity/lineage.js` | 160 | Name (descends from I_AM — `name.parentNameId`) |
| `being/identity/inheritation.js` | 190 | Name (authority lineage) |
| `being/identity/lookups.js` | 284 | mixed |
| `being/identity.js` | 56 | Name |
| `being/beingId.js` | 56 | Being (keep) |

~2000 lines of keypair / signing / credential / lineage logic — the Name's, by the doctrine name.js itself states — stranded under being/identity/. The grep confirms it: `privateKey`, `ed25519`, `encryptCredential`, `mnemonic` all resolve into being/identity/, **not** name/. name/ holds the *schema* and the session/key surface (keys, mnemonic, passwordKey, login, signingSession); the *birth and credential machinery* is still in being/.

So **being/ is not clean of name-stuff** — it carries the entire identity layer. Completing the split (your open "Being split + birth split"): split `birth.js` into Name-minting (→ name/) and body-forming (kept in being/), and move `credentials.js` / `lineage.js` / `inheritation.js` / `identity.js` to name/. After that, being/ is pure presence and name/ is pure identity, and the two stop overlapping.

## name/ is otherwise clean

name.js's 17 "being" references are all **doctrine comments** explaining the split (what's the Being's vs the Name's) — legitimate, not logic. One small real leak the other way: `name/closure.js` imports `I_AM` from `being/seedBeings.js`. But I_AM is a *Name* (name.js: "I_AM is itself a Name, `_id="i-am"`"), so its constant should live in name/ (a `seedNames.js`), not being/seedBeings.js.

## The inherent asymmetries (keep — do NOT force symmetric)

Each kind's own nature, justified:
- **being/** — presence + the being-tree: position, positionAddress, pullBack, seedBeings, seedDelegates.
- **space/** — the tree machinery (the richest): ancestorCache, heavenLineage, heavenSpaces, members, ownership, source, spaceCircuit, spaceLocks, threads, factory, extensionScope.
- **matter/** — content + types: anchor, casSweep, classify, contentStore, coordBounds, types.
- **name/** — the crypto/identity: bip39Words, keys, login, mnemonic, passwordKey, signingSession.
- **history/** — the branch machinery: branchCreation, historyPath, historyScope, historyRegistry, projection, resetReels.

Symmetry is for the **core** (schema/reducer/ops/catalog/id). The periphery is each kind being itself.

## The shared host (materials/*.js, not in a kind)

The generic machinery every kind runs on — the host-side of the wordless engine: `projections.js` (loadOrFold), `reducers.js` + `reducerHelpers.js` (the reducer registry + shared fold), `qualities.js` (the qualities-Map API), `redact.js`, `ref.js` + `refWalker.js`, `_targetShape.js`, `doCeiling.js`. Correctly outside the kinds — it's the loom, not a noun.

## Cleanup plan, by risk

1. **Safe, now:** dead comments per file; this audit; write down the `<kind>Id.js` and `closure.js` rules so the absences read as intentional.
2. **Medium — needs an import sweep + a go:** choose the catalog naming convention and rename the five to it; give name/ and history/ an `ops.js` (or an in-folder note); move I_AM's constant to a name/ home.
3. **Big — needs design + a go:** complete the Name/Being split (split `birth.js`; move credentials/lineage/inheritation/identity → name/). This is the one that actually makes "no name stuff in being" true — a real refactor (birth is the hot path), not a rename.

The symmetric ceiling: every kind reads as `<kind>.js` + `reducer.js` + `ops.js` + one catalog + (optional) `<kind>Id.js`, then its own nature — and each folder is the visible host of its word.
